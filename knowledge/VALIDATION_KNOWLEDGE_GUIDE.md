# VALIDATION KNOWLEDGE GUIDE

**Version:** 1.0
**Date:** 2026-06-08
**Purpose:** Operational guide to the validation pipeline. Companion to VALIDATION_PIPELINE_SPECIFICATION.md and VALIDATION_CHECKLIST.md — explains validation philosophy, categories, workflow, and how to interpret results.

---

## 1. WHY VALIDATION EXISTS

Validation exists because **the cost of releasing invalid canon is higher than the cost of blocking a release.** A bot that contains rejected canon, authority violations, or untraceable content damages the entire project's credibility.

The validation_engine is not a suggestion engine. It is a **mandatory gate.** No output — bot, lorebook, API response, or trace report — is released without passing validation.

---

## 2. VALIDATION PHILOSOPHY

### The Four Principles

1. **Canon supremacy:** The database/ is the Single Source of Truth. All output must match it. Period.
2. **Authority respect:** Data comes from exactly one owning authority. Cross-contamination is a violation.
3. **Traceability:** Every output element must have a complete provenance chain back to a database file.
4. **Layer separation:** Canon layers (Active, Historical, Cultural, Deferred, Candidate) must never be mixed in output.

### What Validation Does NOT Do

- It does not improve or enhance content
- It does not fill in missing data
- It does not make editorial judgments about quality (only compliance)
- It does not modify content to make it pass

Validation is binary: **compliant or non-compliant.**

## 3. VALIDATION CATEGORIES

### 3.1 Cross-Reference Validation (CR)

**Purpose:** Verify all entity references resolve to existing database/ records.

| Check | What It Verifies |
|-------|-----------------|
| CR-001 | Character references → existing C_*.md files |
| CR-002 | Family references → existing F_*.md files |
| CR-003 | Visual references → existing W_*.md files |
| CR-004 | Institution references → existing I_*.md files |
| CR-005 | Experience references → existing Ex_*.md files |
| CR-101 | Parent-child claims match F_Parent_Child.md |
| CR-102 | Marriage claims match F_Marriages.md |
| CR-103 | Sibling claims computable from parent-child records |
| CR-104 | No circular genealogical references |
| CR-201 | Visual claims match W_Visual_Baseline.md |
| CR-202 | Visual inheritance matches W_Visual_Inheritance.md |
| CR-301 | Character names match identity records |
| CR-302 | Personality claims match psychology records |

**When it fails:** Output contains references to entities that don't exist, or claims that contradict authority records.

### 3.2 Canon-Layer Validation (CL)

**Purpose:** Verify correct classification and separation of canon layers.

| Check | What It Verifies |
|-------|-----------------|
| CL-001 | Every element has exactly one canon layer |
| CL-002 | Active not mixed with Cultural |
| CL-003 | Historical properly classified |
| CL-004 | Cultural clearly marked as non-factual |
| CL-005 | Deferred not treated as Active |
| CL-006 | Candidate not treated as approved |
| CL-201 | No rejected canon (Valeria, supernatural, etc.) |

**When it fails:** Layers are mixed, rejected content appears, or classification is wrong.

### 3.3 Authority Ownership Validation (AO)

**Purpose:** Verify data comes from the correct authority layer.

| Check | What It Verifies |
|-------|-----------------|
| AO-001 | Character data from Character Authority |
| AO-002 | Genealogy data from Family Authority |
| AO-003 | Visual data from Visual Authority |
| AO-004 | Scenario data from Experience Authority |
| AO-101 | No engine claims ownership of data it doesn't own |
| AO-104 | No cross-authority contamination |

**When it fails:** Data is present but sourced from the wrong authority.

### 3.4 Traceability Validation (TR)

**Purpose:** Verify every output element has a complete provenance chain.

| Check | What It Verifies |
|-------|-----------------|
| TR-001 | Every element has source file path |
| TR-002 | Source paths point to existing files |
| TR-003 | No orphaned output without canonical source |
| TR-101 | Provenance chain is complete (output → engine → file → authority) |
| TR-102 | Canon version is recorded |

**When it fails:** Output exists but can't be traced back to a database record.

## 4. VALIDATION WORKFLOW

### 4.1 Automated Validation (Engine-Level)

```
Output generated by any engine
    ↓
Receives: data, source_engine, validation_type
    ↓
All four validation categories run:
    Cross-reference → Canon-layer → Authority → Traceability
    ↓
All pass → status: 'valid' → Output released
Any fail → status: 'invalid' → Rejected + error report
Borderline → status: 'warning' → Released with warning log
```

### 4.2 Manual Validation (Human-Level)

For human reviewers, the VALIDATION_CHECKLIST.md provides the 47-check operational guide:

1. Execute checks in order
2. On first FAIL, STOP and report
3. Do not continue until FAIL is resolved
4. Document all findings

### 4.3 Validation Types

| Type | Scope | When Used |
|------|-------|-----------|
| `cross_reference` | Entity resolution only | Quick checks during development |
| `canon_layer` | Layer classification only | Lorebook generation |
| `authority_ownership` | Authority boundaries only | Export formatting |
| `traceability` | Provenance only | Audits |
| `full` | All categories | Release gate (mandatory) |

## 5. PASS / WARNING / FAIL INTERPRETATION

### PASS (valid)
- All applicable checks passed
- Output is compliant with all authority and canon requirements
- Safe to release

### WARNING
- Output is technically compliant but has borderline characteristics
- Example: An entry that BLACTIVELY activates but could potentially conflict with another entry under specific conditions
- Released with warning log for future review
- Does not block release but flags for attention

### FAIL (invalid)
- One or more checks failed
- Output is NOT compliant
- Release is BLOCKED
- Detailed error report is generated
- Must be fixed and re-validated before release

### Common Failure Modes

| Failure | Cause | Fix |
|---------|-------|-----|
| Entity not found | Reference to non-existent record | Verify ID exists in database/ |
| Relationship mismatch | Family claim doesn't match records | Verify against F_Parent_Child.md or F_Marriages.md |
| Layer violation | Cultural content in Active export | Reclassify or add disclaimer |
| Authority violation | Data from wrong authority | Route query to correct authority |
| Rejected canon | Prohibited content detected | Remove content; reference ADR-006 rejected list |
| Missing traceability | No source attribution | Add provenance chain |

## 6. VALIDATION IN THE EXPORT PIPELINE

### Bot Export Validation Flow

```
Data collected from engines
    ↓
validation_engine(full)
    ├─ Cross-reference: All IDs resolve?
    ├─ Canon-layer: All content Active?
    ├─ Authority: All data from correct authority?
    ├─ Traceability: All fields sourced?
    └─ Rejected canon: No prohibited content?
    ↓
PASS → Format → Attach provenance → Release
FAIL → Reject → Error report → Fix source → Retry
```

### Lorebook Export Validation Flow

```
For each entry:
    Generate from database record
    ↓
    validation_engine(cross_reference + canon_layer)
    ↓
    PASS → Add to lorebook
    FAIL → Skip + log error
    ↓
On complete file:
    validation_engine(traceability)
    ↓
    PASS → Write file
    FAIL → Reject file → Fix source entries → Retry
```

## 7. COMMON VALIDATION MISTAKES

### Mistake 1: Treating Warnings as Acceptable Long-Term
**Problem:** Warnings accumulate. 50 warnings in a lorebook means 50 borderline entries that could cause issues.
**Fix:** Investigate every warning. If it's a true false positive, document why. If it's a real issue, fix it.

### Mistake 2: Validating Only at Release
**Problem:** Finding validation errors at release time means rework at the worst possible moment.
**Fix:** Validate early and often. Validate during data collection, during formatting, and at release.

### Mistake 3: Manually Overriding Validation
**Problem:** "I know this is correct even though validation says otherwise." Manual overrides undermine the entire system.
**Fix:** If validation is wrong, fix the validation rule. If the data is wrong, fix the data. Never override.

### Mistake 4: Inconsistent Validation Scope
**Problem:** Running `cross_reference` only when `full` is required. Missing authority violations because the wrong validation type was used.
**Fix:** Always use `full` validation for release. Use narrower types only during development.

## 8. OPERATIONAL CHECKLIST (Summary)

For the complete 47-check checklist, see VALIDATION_CHECKLIST.md. The summary categories are:

1. **Cross-Reference Validation** (CR-001 through CR-304) — 17 checks
2. **Canon-Layer Validation** (CL-001 through CL-205) — 15 checks
3. **Authority Ownership** (AO-001 through AO-104) — 8 checks
4. **Traceability** (TR-001 through TR-103) — 7 checks

**Total: 47 checks. All must PASS for release.**

### Detailed Check Summary

#### Cross-Reference Validation — 17 Checks

| ID Range | Focus | Key Checks |
|----------|-------|-----------|
| CR-001 to CR-005 | Entity resolution | All IDs resolve to existing records |
| CR-101 to CR-104 | Relationship consistency | Parent-child, marriage, sibling claims match |
| CR-201 to CR-202 | Visual consistency | Visual claims match Visual Authority |
| CR-301 to CR-304 | Identity consistency | Names, personality match character records |

#### Canon-Layer Validation — 15 Checks

| ID Range | Focus | Key Checks |
|----------|-------|-----------|
| CL-001 to CL-006 | Layer classification | Exactly one layer per element; no mixing |
| CL-201 to CL-205 | Rejected canon | No prohibited content (Valeria, supernatural, etc.) |

#### Authority Ownership Validation — 8 Checks

| ID Range | Focus | Key Checks |
|----------|-------|-----------|
| AO-001 to AO-004 | Domain ownership | Data from correct authority |
| AO-101 to AO-104 | Engine boundaries | No cross-authority contamination |

#### Traceability Validation — 7 Checks

| ID Range | Focus | Key Checks |
|----------|-------|-----------|
| TR-001 to TR-003 | Source attribution | Every element has source file + path |
| TR-101 to TR-103 | Provenance chain | Complete chain from output to authority |

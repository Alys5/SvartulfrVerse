## Purpose

Define the **Canonical Runtime Model** — the logical data pipeline that transforms repository records into platform-ready artifacts (JanitorAI cards, lorebook entries, generic JSON).

This document **does NOT define code**. It defines the **runtime abstraction** that code will implement:

```
Input Repository
       ↓
Character Loader
       ↓
Context Compiler
       ↓
Retrieval Engine
       ↓
Export Adapter
       ↓
JanitorAI Artifact
```

**Non-negotiable constraint:** This runtime targets JanitorAI as primary platform. JanitorAI has no persistent runtime, no shared JS modules, no state manager. The "runtime" is a **prompt orchestration model** — structured prompt assembly, lorebook retrieval, trigger-based state tracking, and ES5 sandbox scripting.

---

## JanitorAI Runtime Constraints

### What Exists

| Capability | Details |
|------------|---------|
| System Prompt | First text block sent to LLM. Static per session. |
| Personality | Character identity + behavior rules. |
| Memory | 1-2 short paragraphs injected before each response. |
| Scenario | Scene context, default setup. |
| Advanced Post-History Instructions | System-level behavioral constraints injected after messages. |
| Lorebook | Trigger-keyed entries injected on keyword match. ES5 scripting available. |
| Recent Messages | Sliding window of conversation history. |
| ES5 Sandbox | Limited JavaScript for conditional logic in lorebook entries. |

### What DOES NOT Exist

| Absence | Implication |
|---------|-------------|
| Persistent state between Sessions | State must be encoded in lorebook keys or session metadata. |
| Shared JS modules | All logic must be self-contained in prompts or ES5 snippets. |
| Database connections | All data must be pre-compiled into prompts/lorebook entries. |
| Token counting API | Token budget is estimated, not measured programmatically. |
| Runtime authority resolution at inference time | Authority conflicts must be PRE-resolved in the prompt. |

### Prompt Assembly Order (JanitorAI)

```
┌─────────────────────────────────────────┐
│ 1. System Prompt                        │ ← Always first. Authority constraints.
│ 2. Personality                          │ ← Character identity + voice.
│ 3. Memory / Creator Notes              │ ← Session state + creator context.
│ 4. Lorebook Entries (triggered)         │ ← Injected on keyword match.
│ 5. Scenario                             │ ← Experience/scene context.
│ 6. Recent Messages                      │ ← Conversation history.
│ 7. Advanced Post-History Instructions   │ ← Behavioral guardrails.
│ 8. ES5 Script Execution                 │ ← Conditional logic (lorebook only).
└─────────────────────────────────────────┘
```

Data from ALL authorities must be **pre-merged** into these 7 injection points. The LLM sees ONE stream — there is no "authority separation" at inference time. All resolution happens at compile time.

## Pipeline Architecture

### Phase 0: Input Repository

**Source:** `database/` (canonical SSoT per E-17.0)

**Input types:**

| Type | Path Pattern | Authority |
|------|-------------|-----------|
| Character | `database/characters/C_*.md` | Character Authority |
| Family | `database/families/F_*.md` | Family Authority |
| Visual | `database/visuals/V_Visual_*.md` | Visual Authority |
| World | `database/worlds/W_*.md` | World Authority |
| Location | `database/locations/L_*.md` | World Authority |
| Institution | `database/institutions/I_*.md` | World Authority |
| Experience | `database/experiences/Ex_*.md` | Experience Authority |
| Institution | `database/institutions/I_*.md` | World Authority |
| Historical | `database/historical/HC_*.md` | Historical Canon |

**Consumption rule:** Records are read ONCE at compile time. The runtime does NOT re-read files during inference.

---

### Phase 1: Character Loader

**Input:** Character ID + Entry Mode (ADR-008)
**Output:** Character Package

**Logic:**

```
IF entry_mode == CHARACTER:
    LOAD profile from E_17.3 Character Loading Package
    RESOLVE dynasty membership from F_Douglas_Bloodmoon
    RESOLVE family graph from F_Parent_Child + F_Marriages
    RESOLVE visual from V_Visual_Baseline + V_Visual_Inheritance
    RESOLVE world from W_Contemporary
    RESOLVE conditional locations/institutions from context

IF entry_mode == SCENARIO:
    LOAD experience from Ex_*.md
    RESOLVE primary character from experience config
    LOAD character package (as above)
    ATTACH experience rules as Level 2 authority overlay

IF entry_mode == LOCATION:
    LOAD location from L_*.md
    RESOLVE present characters from E_17.3 conditional sets
    LOAD each character package (as above)
    SET location as primary context

IF entry_mode == INSTITUTION:
    LOAD institution from I_*.md
    RESOLVE affiliated characters from E_17.3 conditional sets
    LOAD each character package (as above)
    SET institution as primary context
```

**CONSTRAINT:** Character identity (Level 1) is IMMUTABLE. Experience context (Level 2) modifies behavior overlay only.

---

### Phase 2: Context Compiler

**Input:** Character Package + Active Experience
**Output:** Compiled Context per JanitorAI prompt slots

**Logic:**

```
FOR EACH character package:
    COMPILE Hard Context (identity + visual + current location)
    COMPILE Soft Context (relationships + family + institutions)
    COMPILE Optional Context (historical + extended family + world lore)
    
    CHECK token budget (~1100 tokens dynamic per E-17.2)
    
    IF budget exceeded:
        APPLY compression rules (E-17.2):
            1. Drop Optional context entirely
            2. Compress Soft to 1-line summaries
            3. Truncate historical references
            4. NEVER compress Hard context

RESOLVE authority conflicts per E-17.4:
    Experience context modifies BEHAVIOR only (not identity)
    Visual Authority always applies (never overridden)
    Character identity is IMMUTABLE
```

**Output mapping to JanitorAI prompt slots:**

| JanitorAI Slot | Data Source E_17.2 Tier | Priority |
|----------------|------------------------|----------|
| System Prompt | Authority constraints + World Rules + Behavioral guardrails | Always |
| Personality | Hard Context (identity + voice + speech profile) | Always |
| Memory | Session state + Ex tracking state + Relationship reminders | Always |
| Scenario | Active Experience (if SCENARIO entry) + Location context | Conditional |
| Lorebook | Soft Context + Optional Context + NPC entries | Triggered |
| Post-History | Authority rejection rules + Canon compliance rules | Always |

---

### Phase 3: Retrieval Engine

**Input:** User message + Current Context State
**Output:** Retrieval Decision (what to load/reload)

**Logic:**

```
STEP 1: Entry Mode Detection (E-17.1 Phase 0)
    Scan user input for character names, locations, institutions, experience triggers
    Apply priority: CHARACTER > SCENARIO > LOCATION > INSTITUTION
    Default: CHARACTER

STEP 2: Intent Detection (E-17.1 Phase 1)
    Classify input: SCENE_START, CHARACTER_ACTION, DIALOGUE, NARRATIVE, SETTING_CHANGE, META, EXPERIENCE_TRIGGER
    If EXPERIENCE_TRIGGER matched AND no higher-priority intent → activate experience

STEP 3: Entity Detection (E_17.1 Phase 2)
    Scan for characters, locations, institutions, NPCs

STEP 4: Context Matching (E_17.1 Phase 3)
    Check: Is this entity already in current context?
    If NO → mark for Conditional Load
    If experience ended → deactivate experience, return to CHARACTER entry

STEP 5: Lorebook Retrieval
    FOR EACH active lorebook entry:
        CHECK trigger conditions
        IF trigger matched AND not recently injected → mark for injection
        IF token budget exceeded → skip Optional-priority entries
```

**CONSTRAINT:** The Retrieving Engine operates on PRE-COMPILED lorebook entries. It does NOT read `database/` at this phase. All data must already be in lorebook format.

---

### Phase 4: Export Adapter

**Input:** Compiled Context + Lorebook Entries
**Output:** JanitorAI Character Card + Lorebook JSON + Provenance Metadata

**Logic:**

```
BUILD Character Card:
    name = character.identity.name
    description = composite(visual + personality_density)
    personality = Hard Context personality section
    first_mes = generate_from(personality + scenario_context)
    mes_example = generate_from(speech_profile + canonical_dialogue)
    system_prompt = Compile system prompt
    creator_notes = memory + backstory_summary
    post_history_instructions = authority_constraints
    tags = [canon_layer, dynasty, character_id, "SvartulfrVerse"]
    extensions.svartulfrverse = provenance_metadata (per EXPORT_MAPPING_MATRIX 4.1)

BUILD Lorebook Entries:
    FOR EACH Soft/Optional context item:
        GENERATE entry with trigger keys
        ASSIGN priority (Soft > Optional)
        SET injection budget per entry
        TAG with canon layer + source + authority

BUILD Provenance:
    extensions.svartulfrverse.canon_id = character_record_id
    extensions.svartulfrverse.canon_layer = record.canon_layer
    extensions.svartulfrverse.provenance = [source_file_paths]
    extensions.svartulfrverse.authority = record.authority_domain
    extensions.svartulfrverse.visual_classification = record.visual_class
    extensions.svartulfrverse.dynasty = record.dynasty
    extensions.svartulfrverse.generation_timestamp = ISO_NOW()
    extensions.svartulfrverse.canon_version = "1.0"
```

## State Model

### Active Session State

Tracked via lorebook keys (JanitorAI persistent storage mechanism):

```json
{
  "entry_mode": "CHARACTER | SCENARIO | LOCATION | INSTITUTION",
  "active_character": "C_Jasper",
  "active_experience": "Ex_DJFrequency | null",
  "experience_chapter": "I-IX | null",
  "experience_beats_completed": ["beat_1", "beat_2"],
  "experience_beats_pending": ["beat_3", "beat_4", "..."],
  "current_location": "L_RooftopSet | null",
  "characters_present": ["C_Jasper"],
  "common_language": "English",
  "speech_profile_active": true
}
```

### State Transitions

```
SCENARIO_START:
    trigger: user input matches experience trigger
    guard: no conflicting experience already active
    effect: activate Ex_*.md, set entry_mode=SCENARIO, load Chapter I

SCENARIO_BEAT_COMPLETE:
    trigger: chapter closing condition satisfied
    guard: chapter verified by context compiler
    effect: add beat to completed list, advance chapter counter

SCENARIO_END:
    trigger: final chapter completed OR user triggers exit
    guard: none — user can exit anytime
    effect: deactivate experience, clear state, return to CHARACTER entry

LOCATION_CHANGE:
    trigger: user moves to new location
    guard: new location exists in conditional load set
    effect: update current_location, reload conditional data
```

## Success Criteria

The Runtime Model is complete when:

- [ ] Every JanitorAI prompt slot has a defined data source from E-17.0-E-17.4
- [ ] The pipeline has ZERO authority hierarchy duplication
- [ ] Session state can be fully expressed in a JanitorAI-compatible lorebook key
- [ ] State transitions cover SCENARIO_START, BEAT_COMPLETE, SCENARIO_END, LOCATION_CHANGE
- [ ] Export Adapter produces provenance-compliant artifacts per EXPORT_MAPPING_MATRIX §4
- [ ] Token budget model (E-17.2) is applied at exactly ONE point in the pipeline

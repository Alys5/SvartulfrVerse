## Section 1: Runtime Reality

### 1.1 JanitorAI Runtime Environment

JanitorAI is a prompt-mediated chat interface. The platform assembles a prompt from user-configured fields, sends it to an LLM, and streams the response back. There is no persistent application runtime, no database access, no server-side execution, and no cross-script memory at the model level.

**Observed architecture (from official docs + community guides):**

```
JanitorAI Platform Assembly:
│
├── System Prompt / Advanced Prompt  ← First text block (highest authority)
├── Personality                      ← Character identity + behavior
├── Memory / Creator Notes           ← Session context + creator guidance
├── Lorebook Entries (triggered)     ← Keyword-matched knowledge injection
├── Scenario                         ← Scene context + framing
├── Example Dialogue                 ← Voice training patterns
├── Chat Memory                      ← Recent conversation history
└── [User Message]                   ← Current turn input
```

**Source basis:** JanitorAI official help (Advanced Prompting 101), Scripts Guide, JanitorAI Best Practices (JANITORAI_AND_ROLEPLAY_BEST_PRACTICES §6), community tutorial (jaitutorial.uwu.ai).

### 1.2 Officially Documented Platform Behavior

| Behavior | Source |
|----------|--------|
| System prompt has highest instruction authority | JanitorAI Advanced Prompting 101 |
| Character card fields are injected into the prompt in fixed order | JanitorAI Best Practices §2 |
| Lorebooks are keyword-triggered retrieval mechanisms | JanitorAI Scripts Guide, Icehellionx Advanced Lorebook |
| Scripts run in a sandboxed ES5 environment | JanitorAI Scripts Guide |
| Chat Memory preserves recent conversation context | JanitorAI Chat Memory article |
| Author's Notes / Creator Notes inject mid-conversation context | JanitorAI Best Practices §6.2 |
| Post-history instructions appear after conversation history | JanitorAI Best Practices §6.3 |

### 1.3 Community-Observed Behavior

| Behavior | Source |
|----------|--------|
| Example Dialogue is the strongest voice-training mechanism | Consensus across all major guides (JANITORAI_BEST_PRACTICES §2.5, §4) |
| Lorebook entries that repeat card content waste tokens | All major guides (consensus) |
| Personality drift is the #1 quality issue in long chats | JANITORAI_BEST_PRACTICES §2.5 |
| Lorebook entries should be 40-120 tokens for efficiency | SillyTavern docs adopted by JanitorAI community |
| Post-history instructions have very strong model-following effect | JANITORAI_BEST_PRACTICES §6.3 |
| Opening message tone sets user expectations for entire session | JANITORAI_BEST_PRACTICES §3.1 |

### 1.4 SvartúlfrVerse Architectural Decisions

| Decision | Rationale |
|----------|-----------|
| Identity in Personality, not Lorebooks | Identity must be available from turn zero. Retrieval is unreliable for core identity. |
| Voice in Example Dialogue, not Personality | Example Dialogue is the strongest voice-training mechanism. Personality describes behavior; examples teach voice. |
| Experience context in Scenario, not Lorebooks | Active scene context is permanent per session, not retrieval-dependent. |
| Extended knowledge in Lorebooks, not Card | Lorebooks are retrieval systems. Moving extended content out of the card reduces token pressure. |
| Authority constraints in System Prompt, not Lorebooks | Authority rules must be injection-independent (E_17.4). |

### 2.2 Behavioral Influence Classification

| Classification | Meaning |
|---------------|---------|
| **Runtime Critical** | Directly controls core identity and behavioral rules. Available from turn zero. Cannot depend on retrieval. |
| **Runtime Influential** | Strongly shapes behavior and voice. Available from turn zero or first beat. |
| **Runtime Supportive** | Adds depth and context. Retrieval-dependent. May not always be active. |
| **Runtime Cosmetic** | No behavioral influence. Presentation layer only. |

## Section 3: JanitorAI Constraints

### 3.1 Platform Capabilities

| Capability | Details | Source |
|------------|---------|--------|
| System Prompt / Advanced Prompt | Free-text field. First text block sent to LLM. Highest authority. | JanitorAI Advanced Prompting 101 |
| Personality | Free-text field. Character identity + behavior definition. | JanitorAI Best Practices §2 |
| Memory / Creator Notes | Short text field. Injected before each response as session context. | JanitorAI Best Practices §6.2 |
| Scenario | Free-text field. Scene context, framing, active state. | JanitorAI Best Practices §3 |
| Example Dialogue | Fixed format: `{{char}}:` / `{{user}}:` exchanges. 3-5 exchanges recommended. | JanitorAI Best Practices §4 |
| Initial Message | Character's first message. Establishes tone and dynamic. | JanitorAI Best Practices §3.1 |
| Lorebook (Scripts → Lorebook) | keyword-triggered entries. ES5 scripting available for conditional logic. | JanitorAI Scripts Guide, Icehellionx guides |
| Chat Memory | Platform-managed recent message buffer. Depth configurable. | JanitorAI Chat Memory article |
| Bio (HTML) | Character page description. HTML formatting supported. | JanitorAI platform UI |
| Tags | Array of strings for search/organization. | JanitorAI platform UI |
| Character Image | PNG/JPG avatar. Displayed on character page and chat. | JanitorAI platform UI |

### 3.2 Platform Absences (Confirmed)

| Absence | Implication |
|---------|-------------|
| No persistent application state between messages | State must be encoded in prompt or lorebook keys |
| No database access at inference time | All data must be pre-compiled into prompt fields |
| No server-side execution | All logic is prompt-mediated or ES5-sandboxed |
| No repository access during chat | Compiler runs OFFLINE, generates artifacts, then upload |
| No cross-script memory between sessions | Each session is independent; continuity via prompt fields |
| No token counting API | Token budget estimated by compiler, not measured at runtime |
| Bio does not meaningfully influence runtime | No identity/voice/behavior content in bio |

## Section 4: Export Package Structure

### 4.1 Canonical Export Package

```
Bot/
├── personality.md           ← Runtime Critical — identity + behavior + voice rules
├── scenario.md              ← Runtime Influential — active experience + scene context
├── initial_message.md       ← Runtime Influential — opening hook + tone setting
├── example_dialogue.md      ← Runtime Influential — voice training patterns
├── system_prompt.md         ← Runtime Critical — authority constraints + world rules
└── metadata.md              ← Informational — provenance + canonical traceability

Lorebooks/
├── LB_Core.json             ← Active from session start (character essentials)
├── LB_World.json            ← Triggered by location/institution keywords
├── LB_Family.json           ← Triggered by family member keywords
├── LB_Character_Jasper.json ← Triggered by character-specific keywords
└── LB_Experience_DJF.json   ← Triggered by experience-specific keywords

Scripts/
└── (optional)               ← ES5 runtime scripts for state tracking

Bio/
└── bio.html                 ← Runtime Cosmetic — presentation only
```

### 4.2 Artifact Specification

#### personality.md

| Field | Value |
|-------|-------|
| Runtime Importance | CRITICAL |
| Behavioral Influence | Controls identity, personality, behavioral rules, speech patterns |
| Authority Level | Character Authority (Level 1) + ADR-009 Speech Profile |
| Source Data | entity.identity, entity.personality, entity.speech_profile |
| Compilation Rules | Transform structured data into prose descriptors. Include: (1) core identity, (2) personality traits with behavioral anchors, (3) speech patterns with specific examples, (4) boundaries and limits, (5) motivations and fears. NEVER include compiled text that belongs in Lorebooks. NEVER use retrieval-dependent content. |
| Token Budget | ~300-500 tokens (hard context) |

**Compilation rules (from JANITORAI_BEST_PRACTICES §2):**
- 3+ specific behavioral patterns (not trait labels)
- 1 clear motivation driving behavior
- 1 clear flaw creating friction
- Positive framing ("does this") over negative ("doesn't do that")
- Concise prose. Cut 20% test applies.

#### scenario.md

| Field | Value |
|-------|-------|
| Runtime Importance | HIGH |
| Behavioral Influence | Controls current situation, active experience, immediate context |
| Authority Level | Experience Authority (Level 2) + World Authority (Level 3) |
| Source Data | Active experience record + current location + session state |
| Compilation Rules | Generate from: (1) experience narrative framework, (2) active beat entry conditions, (3) current location atmosphere, (4) present characters list, (5) world rules active for this scenario. NEVER repeat identity content from personality.md. |
| Token Budget | ~200-300 tokens |

**Constraint:** Scenario defines what is happening NOW. It does not redefine who the character is. Identity is Personality's domain.

#### initial_message.md

| Field | Value |
|-------|-------|
| Runtime Importance | HIGH |
| Behavioral Influence | Sets tone, framing, first interaction dynamic |
| Authority Level | E_17.4 authority hierarchy + active Scenario + Beat entry conditions |
| Source Data | Active experience chapter I + first beat + scenario context |
| Compilation Rules | Generate: (1) character POV (first person or third per character), (2) action-in-progress (not waiting/empty greeting), (3) implicit question or hook for user response, (4) voice consistency with example_dialogue.md. Target under 250 tokens. NEVER write "Hi" as opening. |
| Token Budget | ~100-250 tokens |

**Compilation rules (from JANITORAI_BEST_PRACTICES §3.1):**
- Start in media res when possible
- Include a choice-demanding prompt
- Match emotional state to scene (not to idealized default)
- Leave room for user to define their entrance

#### example_dialogue.md

| Field | Value |
|-------|-------|
| Runtime Importance | HIGH |
| Behavioral Influence | Strongest voice-training mechanism. Model mirrors cadence, vocabulary, formatting. |
| Authority Level | ADR-009 Speech Profile + entity.personality.communication_style |
| Source Data | speech_profile.register, speech_profile.slang_profile, speech_profile.speech_traits, canonical character dialogue |
| Compilation Rules | Generate 3-5 exchanges in `{{char}}:` / `{{user}}:` format. Each exchange demonstrates a DIFFERENT aspect: (1) default voice, (2) emotional range, (3) humor/quickness, (4) depth/vulnerability, (5) boundary/assertiveness. Total: 100-300 tokens. |
| Token Budget | ~100-300 tokens |

**Constraint (from JANITORAI_BEST_PRACTICES §2.5 §4):**
- This is the PRIMARY voice-teaching mechanism
- Scenario does NOT teach voice. Personality describes behavior. Examples TEACH voice.
- Pattern match quality > quantity

#### system_prompt.md

| Field | Value |
|-------|-------|
| Runtime Importance | CRITICAL |
| Behavioral Influence | Highest instruction authority. Model follows system prompt above all other fields. |
| Authority Level | E_17.4 Runtime Authority Resolution + ADR-000 Runtime Baseline + ADR-008/009 Runtime Constraints |
| Source Data | E_17.4 rejection rules, ADR-000 world rules, ADR-008 entry mode rules, ADR-009 language state |
| Compilation Rules | Generate: (1) narrative perspective rules (never speak for {{user}}), (2) authority rejection rules (identity immutable, visual always applies), (3) world rules (human-only, no supernatural), (4) OOC handling rules, (5) POV and formatting rules. Keep concise. Hard constraints only. |
| Token Budget | ~150-250 tokens |

#### metadata.md (extensions.svartulfrverse)

| Field | Value |
|-------|-------|
| Runtime Importance | NONE (informational) |
| Behavioral Influence | None. Not sent to model. Stored in character card metadata. |
| Authority Level | EXPORT_MAPPING_MATRIX §4 traceability requirements |
| Source Data | entity.provenance |
| Compilation Rules | Map entity.provenance + compilation timestamp into JSON object per EXPORT_MAPPING_MATRIX §4.1. |
| Token Budget | N/A (metadata field, not prompt) |

#### Lorebook Entries (.json)

| Field | Value |
|-------|-------|
| Runtime Importance | MEDIUM |
| Behavioral Influence | Adds knowledge depth. Influences behavior only when triggered. |
| Authority Level | Respective authorities per entry (Family, World, Experience, etc.) |
| Source Data | E_17.3 conditional load set + BOT_SEARCH_SPECIFICATION entry structure |
| Compilation Rules | Each entry: (1) unique ID, (2) trigger keywords (specific, non-overlapping), (3) content (40-120 tokens, standalone), (4) canon layer tag, (5) source reference, (6) priority. NO entry repeats Personality or Scenario content. |
| Token Budget | 40-120 tokens per entry |

**Constraint (from BOT_SEARCH_SPECIFICATION + JANITORAI_BEST_PRACTICES §5):**
- Lorebooks are retrieval systems. They ARE NOT identity systems, voice systems, or scenario systems.
- Entries must be standalone understandable.
- Keywords must avoid over-triggering (broad) and under-triggering (narrow).

#### bio.html

| Field | Value |
|-------|-------|
| Runtime Importance | NONE (cosmetic) |
| Behavioral Influence | ZERO. Does not appear in prompt assembly. |
| Authority Level | NONE |
| Source Data | entity.identity.summary + entity.provenance |
| Compilation Rules | Generate HTML-formatted character summary for character page display. Include: character name, brief description (2-3 sentences), dynasty/status. |
| Token Budget | N/A (not in prompt) |

**Explicit rule: bio.html has zero runtime authority. The compiler generates it for presentation purposes only.**

### 5.2 Mapping Rules

1. **Identity fields** (name, age, traits, motivations) → Personality.md. Never in lorebooks.
2. **Visual fields** (hair, eyes, build, height) → Personality.md (appearance section). Described by compiler, not repeated.
3. **Voice fields** (speech profile, register, slang, patterns) → Personality.md (behavior section) + Example Dialogue.md (demonstration).
4. **Experience context** (scenario, active chapter, world rules) → Scenario.md.
5. **Initial state** (first beat, entry condition) → Initial Message.md.
6. **Authority constraints** (baseline rules, rejection rules, world rules) → System Prompt.md.
7. **Extended knowledge** (relationships, backstory, world lore) → Lorebook entries.
8. **Provenance** (canon version, source files, authority) → metadata.md (extensions field).
9. **Presentation** (character page summary) → bio.html.

### 6.3 Compiler Responsibilities

The compiler MUST:

1. **Transform** structured repository data into prose artifacts suitable for LLM consumption.
2. **Resolve** authority conflicts per E_17.4 before generating any artifact. Pre-resolution is mandatory — the model sees ONE stream.
3. **Map** every data element to exactly ONE artifact. No duplication across artifacts (except visual reference in both personality and lorebook when lorebook adds new information).
4. **Inject** all Runtime Critical content into prompt fields that are available from turn zero (Personality, System Prompt, Scenario, Example Dialogue, Initial Message).
5. **Optimize** token usage per JANITORAI_BEST_PRACTICES constraints (cut 20% test, 40-120 tokens per lorebook entry, etc.).
6. **Tag** every lorebook entry with canon layer, authority domain, and source reference.
7. **Generate** provenance metadata with full traceability to source files.
8. **Preserve** user agency in initial_message (open-ended hook, not forced response).

### 6.4 Compiler Restrictions

The compiler MUST NOT:

1. **Alter canon.** Repository data is authoritative. The compiler transforms representation, not content.
2. **Create new canonical content.** All facts must trace to a source record.
3. **Store identity in Lorebooks.** Identity is retrieval-dependent and unreliable for turn-zero availability.
4. **Store voice in Personality alone.** Voice training requires Example Dialogue (strongest mechanism).
5. **Duplicate content across artifacts.** Each artifact has a single purpose. Overlapping content wastes tokens.
6. **Include lorebook-exclusive content in card fields.** Card fields are permanent; lorebook content is triggered. Mixing them defeats the purpose.
7. **Generate bio.html content that influences behavior.** Bio is cosmetic only. Content there is not in the prompt.
8. **Exceed token budgets.** Hard context stays under ~1100 dynamic tokens (E_17.2).

## Section 7: Runtime Principles

These are architectural rules derived from §1–§6 above.

### Principle 1: Identity Lives In Personality

**Source:** E_17.4 Level 1 (Character Identity is IMMUTABLE) + JANITORAI_BEST_PRACTICES §2

Character identity — name, age, core personality, motivations, behavioral rules, speech patterns — lives exclusively in the Personality field.

Identity must NOT be stored in lorebooks. Identity must NOT depend on retrieval. Identity must be available from turn zero.

### Principle 2: Voice Lives In Example Dialogue

**Source:** JANITORAI_BEST_PRACTICES §4 (Example Dialogue is strongest voice-training mechanism)

Example Dialogue is the primary voice-training mechanism. It teaches speech cadence, vocabulary, formatting habits, interaction style, and conversational rhythm.

Do NOT rely on lorebooks to teach voice. Do NOT rely on scenario to teach voice. Voice training belongs primarily in Example Dialogue.

### Principle 3: Context Lives In Scenario

**Source:** E_17.4 Level 2 (Experience) + Level 3 (World)

Scenario contains: current situation, active experience, active relationship baseline, active narrative state, immediate contextual framing.

Scenario defines what is happening now. Scenario does NOT redefine identity (that is Personality's domain).

### Principle 4: Knowledge Lives In Lorebooks

**Source:** BOT_SEARCH_SPECIFICATION §1 + JANITORAI_BEST_PRACTICES §5 (Three-Layer Lorebook Model)

Lorebooks contain: world knowledge, location knowledge, family knowledge, event knowledge, experience knowledge.

Lorebooks ARE retrieval systems. Lorebooks are NOT identity systems. Lorebooks are NOT voice systems. Lorebooks are NOT scenario systems.

### Principle 5: Initial Message Starts The Experience

**Source:** JANITORAI_BEST_PRACTICES §3.1 (Opening Messages)

Initial Message exists to: establish tone, establish framing, establish first interaction, reinforce scenario assumptions.

It is onboarding content. It is not the primary identity authority (Personality is).

Best practices: under 250 tokens, in media res when possible, includes choice-demanding hook, leaves user agency.

### Principle 6: Chat Memory Preserves Long-Term Context

**Source:** JANITORAI_BEST_PRACTICES §10 (Long-Term Memory)

Chat Memory preserves important facts, maintains continuity, reinforces ongoing state.

Chat Memory supplements runtime memory. Chat Memory does NOT replace Personality. Chat Memory does NOT replace Scenario.

Creator Notes can reinforce continuity: "Continuing from last session. {{char}} and {{user}} just..."

### Principle 7: Bio Is Presentation Only

**Source:** Platform observation — bio.html does not appear in prompt assembly.

bio.html has ZERO runtime authority. It exists solely for presentation, discoverability, and user information. It does not influence model behavior.

### Principle 8: Authority Constraints Live In System Prompt

**Source:** E_17.4 Runtime Authority Resolution + JanitorAI Advanced Prompting 101

System prompt has the HIGHEST instruction authority. Authority constraints (identity immutable, visual always applies, world rules, rejection rules) must be in system prompt because they cannot depend on retrieval.

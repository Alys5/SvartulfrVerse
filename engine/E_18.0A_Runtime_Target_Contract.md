## Section A: The Runtime Reality Baseline

### A.1 — What JanitorAI IS

JanitorAI is a **prompt-mediated chat interface**. The platform assembles a text prompt from user-configured fields, sends it to an LLM API, and streams the response back to the user. There is no application runtime between messages. The LLM is stateless between turns — everything it "knows" must come from the current prompt assembly.

**JanitorAI prompt assembly order:**

```
┌──────────────────────────────────────────────────────┐
│               JANITORAI PROMPT ASSEMBLY              │
│              (First → Highest Authority)              │
├──────────────────────────────────────────────────────┤
│                                                      │
│  1. System Prompt / Advanced Prompt                  │  ← Behavioral guardrails
│  2. Personality                                      │  ← Character identity + behavior
│  3. Creator Notes / Chat Memory                      │  ← Session continuity
│  4. Lorebook Entries (triggered by keyword match)    │  ← Retrieval knowledge
│  5. Scenario                                         │  ← Scene context + framing
│  6. Example Dialogue                                 │  ← Voice training patterns
│  7. Initial Message (first turn only)                │  ← Onboarding / hook
│  8. Recent Messages (sliding window)                 │  ← Conversation history
│  9. [User Message]                                   │  ← Current turn input
│                                                      │
│  Optional: ES6-safe JanitorAI Script Execution (sandboxed)          │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**Sources:** JanitorAI official help (Advanced Prompting 101), Scripts Guide, jaitutorial.uwu.ai, JANITORAI_AND_ROLEPLAY_BEST_PRACTICES §6 (Instruction Hierarchy).

### A.2 — What JanitorAI Does NOT Exist

These capabilities are **confirmed absent**. The compiler cannot rely on any of them.

| Absence | Implication |
|---------|-------------|
| No shared JS runtime between sessions | State must be encoded in prompt fields or lorebook keys |
| No persistent application state between messages | Each turn is independent; continuity via injected fields only |
| No database access at inference time | All canonical data must be pre-compiled into prompt artifacts |
| No server-side execution during chat | Compiler runs OFFLINE; uploads artifacts BEFORE session starts |
| No repository access during chat | Read-only prompt assembly from pre-compiled artifacts |
| No cross-script memory or event buses | Scripts operate in isolated sandbox; no inter-script communication |
| No guaranteed script execution order | Scripts trigger independently; no orchestration guarantee |
| No token counting API | Token budget is estimated by compiler, never measured at runtime |
| Bio (HTML) does not appear in prompt assembly | bio.html has zero runtime influence — presentation only |

## Section B: Runtime Artifact Specification & Classification

### B.0 — Classification Taxonomy

Every JanitorAI slot is classified using exactly these four levels:

| Classification | Definition |
|---------------|------------|
| **Runtime Critical** | Breaks the bot if missing or wrong. Controls core identity, non-negotiable rules. Available from turn zero. |
| **Runtime Influential** | Guides the model heavily. Shapes behavior, voice, scene context. Available from turn zero or first beat. |
| **Runtime Supportive** | Provides context when queried or triggered. Retrieval-dependent. May not always be active. |
| **Runtime Cosmetic** | No behavioral impact. Presentation layer only. Does not appear in prompt assembly. |

---

### B.1 — Personality

| Attribute | Value |
|-----------|-------|
| **Runtime Importance** | **Runtime Critical** |
| **Behavioral Influence** | Controls core identity, personality traits, behavioral rules, motivations, boundaries, speech patterns. Strongest identity anchor after System Prompt. |
| **Authority Level** | Character Authority (E_17.4 Level 1 — IMMUTABLE). Speech Profile from ADR-009. |
| **Source Data** | `entity.identity` (name, age, gender, nationality), `entity.personality` (traits, motivations, fears, behavioral patterns), `entity.speech_profile` (register, slang, accent, heritage languages) |
| **Compilation Rules** | Transform structured data into prose descriptors. Include: (1) core identity summary, (2) personality traits with behavioral anchors (not labels), (3) speech pattern descriptions, (4) boundaries and limits, (5) motivations and fears. Apply cut 20% test. Use positive framing. Consult JANITORAI_BEST_PRACTICES §2 for minimum viable character (3 behavioral patterns, 1 motivation, 1 flaw, distinct voice). |
| **Token Budget** | 300-500 tokens (hard context) |
| **Source Reference** | JANITORAI_BEST_PRACTICES §2 (Character Card Design), §2.5 (Behavioral Consistency), §2.6 (Avoiding Over-Description) |

---

### B.2 — Scenario

| Attribute | Value |
|-----------|-------|
| **Runtime Importance** | **Runtime Influential** |
| **Behavioral Influence** | Controls current situation, active experience context, immediate framing, relationship baseline, narrative state. |
| **Authority Level** | Experience Authority (E_17.4 Level 2), World Authority (Level 3), E_17.1 Entry Mode |
| **Source Data** | Active `experience` record (narrative framework, arc structure, world rules), current `location` entity (atmosphere, narrative functions), session state (active chapter, beats completed) |
| **Compilation Rules** | Generate from: (1) experience narrative framework, (2) active beat entry conditions, (3) current location atmosphere, (4) present characters list, (5) active world rules. Use immediate framing — what is happening now. NEVER repeat identity content from Personality.md. |
| **Token Budget** | 200-300 tokens |
| **Source Reference** | JANITORAI_BEST_PRACTICES §3 (Scenario Design) |

---

### B.3 — Initial Message

| Attribute | Value |
|-----------|-------|
| **Runtime Importance** | **Runtime Influential** |
| **Behavioral Influence** | Sets tone, framing, first interaction dynamic. Onboarding content that influences session trajectory. |
| **Authority Level** | E_17.4 authority hierarchy + active Scenario + Beat entry conditions |
| **Source Data** | Active experience Chapter I + first beat entry condition + character voice from speech_profile |
| **Compilation Rules** | Generate: (1) character POV (first or third per character), (2) action-in-progress (not waiting/empty greeting), (3) implicit question or choice-demanding hook, (4) voice consistent with example_dialogue.md. Target under 250 tokens. Start in media res when possible. NEVER write "Hi" as opening. Leave user agency for response. |
| **Token Budget** | 100-250 tokens |
| **Source Reference** | JANITORAI_BEST_PRACTICES §3.1 (Opening Messages) |

---

### B.4 — Example Dialogue

| Attribute | Value |
|-----------|-------|
| **Runtime Importance** | **Runtime Influential** |
| **Behavioral Influence** | Strongest voice-training mechanism in the entire prompt stack. Model mirrors cadence, vocabulary, formatting habits, interaction style, conversational rhythm. |
| **Authority Level** | ADR-009 Speech Profile + `entity.personality.communication_style` |
| **Source Data** | `speech_profile.register`, `speech_profile.slang_profile`, `speech_profile.speech_traits`, `speech_profile.accent_profile`, canonical character dialogue |
| **Compilation Rules** | Generate 3-5 exchanges in `{{char}}:` / `{{user}}:` format. Each exchange demonstrates a DIFFERENT aspect: (1) default voice, (2) emotional range, (3) humor/quickness, (4) depth/vulnerability, (5) boundary/assertiveness. Total: 100-300 tokens. Quality > quantity. Pattern fidelity matters more than coverage. |
| **Token Budget** | 100-300 tokens |
| **Source Reference** | JANITORAI_BEST_PRACTICES §4 (Dialogue Design) |

> **Critical rule:** This is the PRIMARY voice-teaching mechanism. Scenario does NOT teach voice. Personality DESCRIBES behavior. Example Dialogue DEMONSTRATES voice.

---

### B.5 — Chat Memory / Creator Notes

| Attribute | Value |
|-----------|-------|
| **Runtime Importance** | **Runtime Supportive** |
| **Behavioral Influence** | Preserves continuity, reinforces ongoing state, provides session context. Supplements but does NOT replace Personality or Scenario. |
| **Authority Level** | System-generated summary + compiler-initiated Creator Notes |
| **Source Data** | Session state, relationship milestones, experience progression, compiler-generated continuity notes |
| **Compilation Rules** | Compiler generates initial Creator Notes for session start: active context summary, relationship state, experience position. Subsequent summary is platform-managed. Keep under 100 tokens. Strategic placement: scene context + emotional state reminders. |
| **Token Budget** | ~50-100 tokens |
| **Source Reference** | JANITORAI_BEST_PRACTICES §6.2 (Author's Notes), §10 (Long-Term Memory) |

---

### B.6 — Lorebooks

| Attribute | Value |
|-----------|-------|
| **Runtime Importance** | **Runtime Supportive** |
| **Behavioral Influence** | Adds knowledge depth that card cannot hold. Influences behavior only when keywords trigger entry injection. Never active unless triggered. |
| **Authority Level** | Per-entry: Family Authority, World Authority, Experience Authority, Character Authority |
| **Source Data** | E_17.3 Conditional Load set + BOT_SEARCH_SPECIFICATION entry structure + relationship entities + world lore + institutional details |
| **Compilation Rules** | Each entry: (1) unique ID (LB-XXX-NNN), (2) trigger keywords (specific, non-overlapping, include synonyms), (3) content (40-120 tokens, standalone-readable), (4) canon layer tag (active/historical/cultural), (5) source reference, (6) priority for injection ordering. Entries MUST NOT repeat Personality or Scenario content. Entries MUST be understandable without other entries active. |
| **Token Budget** | 40-120 tokens per entry; total lorebook budget ~500 tokens |
| **Source Reference** | BOT_SEARCH_SPECIFICATION §1 (Lorebook Structure) + JANITORAI_BEST_PRACTICES §5 (Lorebook Design) |

> **Critical rule:** Lorebooks ARE retrieval systems. They are NOT identity systems, voice systems, or scenario systems.

### B.7 — System Prompt / Advanced Prompt

| Attribute | Value |
|-----------|-------|
| **Runtime Importance** | **Runtime Critical** |

| **Compilation Rules** | Generate: (1) narrative perspective rules (never describe user's thoughts/feelings, never speak for {{user}}), (2) authority rejection rules (E_17.4 §Resolution Rules table), (3) world rules (human-only, no supernatural, Twin Link only in twin scenarios), (4) OOC handling rules, (5) POV and formatting rules. Keep concise. Hard constraints only. |
| **Token Budget** | 150-250 tokens |
| **Source Reference** | JANITORAI_BEST_PRACTICES §6.3 (Post-History), §6.5 (Instruction Hierarchy), JanitorAI Advanced Prompting 101 |

## Section C: Architectural Axioms (The Immutable Rules)

These are absolute laws of the SvartúlfrVerse runtime. They are derived from Section A (Runtime Reality) and Section B (Artifact Specification). The compiler MUST enforce all of them.

---

### Axiom 1: Identity Lives In Personality

Character identity — name, age, core personality, motivations, behavioral rules, speech profiles — lives exclusively in the Personality field.

- Identity MUST NOT be stored in Lorebooks.
- Identity MUST NOT depend on retrieval mechanisms.
- Identity MUST be available from turn zero (before any lorebook triggers).
- Identity is IMMUTABLE at runtime (E_17.4 Level 1). No other authority can override it.

**Source:** E_17.4 (Character Identity IMMUTABLE) + JANITORAI_BEST_PRACTICES §2 (Card Design)

---

### Axiom 2: Voice Lives In Example Dialogue

Example Dialogue is the primary voice-training mechanism. It teaches speech cadence, vocabulary, formatting habits, interaction style, and conversational rhythm.

- Do NOT rely on Lorebooks to teach voice.
- Do NOT rely on Scenario to teach voice.
- Voice training belongs primarily in Example Dialogue.
- Personality describes behavior patterns; Example Dialogue demonstrates voice patterns.

**Source:** JANITORAI_BEST_PRACTICES §4 (Dialogue is strongest voice-training mechanism)

---

### Axiom 3: Context Lives In Scenario

Scenario defines the active experience, immediate framing, narrative state, and current situation.

- Scenario contains: active experience, active relationship baseline, immediate contextual framing.
- Scenario defines what is happening now.
- Scenario does NOT redefine identity (that is Personality's domain per Axiom 1).

**Source:** E_17.4 Level 2 (Experience Context) + JANITORAI_BEST_PRACTICES §3 (Scenario Design)

---

### Axiom 4: Knowledge Lives In Lorebooks

Lorebooks function purely as retrieval systems for world knowledge, family knowledge, and event knowledge.

- Lorebooks ARE retrieval systems.
- Lorebooks are NOT identity systems (Axiom 1).
- Lorebooks are NOT voice systems (Axiom 2).
- Lorebooks are NOT scenario systems (Axiom 3).
- Lorebooks add depth that the character card cannot hold due to token pressure.

**Source:** BOT_SEARCH_SPECIFICATION §1 + JANITORAI_BEST_PRACTICES §5 (Three-Layer Lorebook Model)

---

### Axiom 5: Initial Message Starts The Experience

Initial Message is onboarding content that establishes tone, framing, and first interaction.

- It exists to: establish tone, establish framing, establish first interaction, reinforce scenario assumptions.
- It is onboarding content, not the primary identity authority (Personality is).
- Best practices: under 250 tokens, in media res when possible, includes choice-demanding hook, leaves user agency.

**Source:** JANITORAI_BEST_PRACTICES §3.1 (Opening Messages)

---

### Axiom 6: Chat Memory Preserves Long-Term Context

Chat Memory exists to preserve important facts, maintain continuity, and reinforce ongoing state.

- It supplements runtime memory.
- It does NOT replace Personality (identity).
- It does NOT replace Scenario (active context).

**Source:** JANITORAI_BEST_PRACTICES §10 (Long-Term Memory & Continuity)

---

### Axiom 7: Bio Is Presentation Only

bio.html has ZERO runtime authority.

- It exists solely for presentation, discoverability, and user information.
- Content placed in bio.html does not influence model behavior.
- It must not contain identity, voice, or behavioral content.

**Source:** Platform observation — bio.html does not appear in JanitorAI prompt assembly

---

### Axiom 8: Authority Constraints Live In System Prompt

System Prompt has the HIGHEST instruction authority in JanitorAI prompt assembly.

- Authority constraints (identity immutable, visual always applies, world rules, rejection rules) MUST be in System Prompt.
- Authority rules are injection-independent — they cannot depend on lorebook triggers.
- Pre-resolution is mandatory: the compiler resolves all authority conflicts (per E_17.4) before generating the System Prompt. The model sees ONE stream.

**Source:** E_17.4 (Runtime Authority Resolution) + JANITORAI_BEST_PRACTICES §6.5 (Instruction Hierarchy)

### F.3 — Compiler Responsibilities (Permitted Transformations)

The compiler MAY:

1. **Transform representation** — Convert structured repository fields (tables, YAML blocks, markdown lists) into prose suitable for LLM consumption.
2. **Resolve authority conflicts** — Apply E_17.4 override chain before generating any artifact. Pre-resolution is mandatory. The model sees ONE merged stream.
3. **Optimize token usage** — Apply E_17.2 compression rules. Cut redundant phrasing. Apply the cut 20% test. Stay within per-artifact token budgets.
4. **Generate provenance** — Map every artifact to its source files. Produce `extensions.svartulfrverse` metadata with full traceability.
5. **Select conditional content** — Apply E_17.3 conditional load sets. Load only records relevant to current entry mode, location, and experience.
6. **Tag everything** — Every lorebook entry gets canon layer, authority, source reference. Every artifact gets token budget annotation.
7. **Preserve user agency** — Initial messages leave open-ended hooks, not forced responses.

### F.4 — Compiler Restrictions (Absolute Prohibitions)

The compiler MUST NOT:

1. **Alter canon** — Repository data is authoritative. The compiler transforms representation, NOT content. If `C_Jasper.md` says Jasper has mint green eyes, the personality block says mint green eyes.
2. **Create new canonical content** — All facts in exported artifacts must trace to a source record. No gap-fillers, no inference, no generation of canonical facts.
3. **Store identity in Lorebooks** — Per Axiom 1. Identity is retrieval-dependent; retrieval is unreliable for turn-zero availability.
4. **Store voice only in Personality** — Per Axiom 2. Voice training requires Example Dialogue — the strongest mechanism. Personality describes behavior; examples demonstrate voice.
5. **Duplicate content across artifacts** — Each artifact has exactly ONE purpose. Overlapping content wastes tokens and creates potential contradiction. Exception: visual attributes may appear in both personality (summary) and lorebook (narrative detail) when lorebook adds new information.
6. **Include lorebook-only content in card fields** — Card fields are permanent; lorebook content is triggered. Mixing them defeats the retrieval architecture.
7. **Generate bio.html content that influences behavior** — Per Axiom 7. Bio is not in the prompt. Do not place identity, voice, or behavioral content there.
8. **Exceed token budgets** — Hard context stays under ~1100 dynamic tokens (E_17.2). Each artifact respects its defined budget. If over budget, compress per E_17.2 §Compression Rules (drop Optional → compress Soft → NEVER compress Hard).

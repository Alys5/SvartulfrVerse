# E-18.1: Canonical Data Schema

**Status:** DRAFT
**Date:** 2026-06-10
**Authority:** Engine Authority (R-007)

## Purpose

Define the **formal data contract** for the SvartúlfrVerse runtime. Every module — Character Loader, Context Compiler, Retrieval Engine, Export Adapter — MUST consume and produce objects conforming to this schema.

**This document is the boundary:**

| Layer Above | This Document | Layer Below |
|-------------|---------------|-------------|
| E_18.0 Runtime Model (pipeline orchestration) | **E_18.1 Schema (data shapes)** | E_18.2+ Code (JS implementation) |

After this document is approved, implementation requires **zero additional architectural decisions**. The code is mechanical.

---

## Design Principles

### Principle 1: Entity-First, Not Domain-First

The schema is organized around **entities** (anything that exists in the world), not **domains** (who owns the data).

```
Domain (ownership):    Character Authority, Family Authority, World Authority, ...
Entity (runtime):     Jasper, Erik, The Verve, KSA, Douglas Estate, Ex_DJFrequency
```

A character, a location, and a faction are all `entity` objects with different `type` values. This schema is extensible: adding `species`, `vehicle`, `species` in the future requires only a new `type` enum value, not a new schema section.

### Principle 2: No Compiled Text in Repository Objects

Repository Layer objects contain **data only**. No pre-compiled prose, no personality blocks, no scenario descriptions. All text generation is the compiler's responsibility.

```
ENTITY.personality.traits        → ["rebellious", "empathic"]   ✅ Data
ENTITY.personality.personality_block → "Jasper is chaotic-good..." ❌ Compiled text
```

### Principle 3: Relationship as First-Class Entity

Relationships are not properties of entities. They are **independent objects** that connect entities. This allows:
- Multiple relationship types between the same pair
- Relationship metadata (history, tension, visibility)
- Runtime relationship creation without modifying entity objects

### Principle 4: Three-Layer Separation

```
Repository Layer   → Source data (from database/ records)
Runtime Layer      → Loader output (active session state)
Export Layer       → Adapter output (JanitorAI prompt slots)
```

Objects MUST NOT cross layers. Repository objects are never passed to the Export Adapter. Runtime objects are never written back to the repository.

## Session State Transitions

Per E_18.0. Formalized:

### SCENARIO_START
```
Precondition:  session.active_experience == null
Trigger:       User input matches experience trigger conditions
Guard:         No conflicting experience already active
Action:        Set session.active_experience = <Ex_ID>
               Set session.entry_mode = "scenario"
               Set session.active_experience.chapter = first chapter
               Set session.active_experience.beats_pending = beats for that chapter
               Load Ex world rules into session
Postcondition: session.active_experience != null
```

### BEAT_COMPLETE
```
Precondition:  session.active_experience != null
               active_beat.status == "in_progress"
Trigger:        Beat closing condition satisfied in conversation
Guard:          Beat verified by context compiler (not hallucinated)
Action:         Move active_beat.id from beats_pending to beats_completed
               Set active_beat.status = "completed"
               If last beat in chapter → advance chapter, load next beats
Postcondition:  beat moved to completed list
```

### SCENARIO_END
```
Precondition:  session.active_experience != null
Trigger:        Final chapter final beat completed
               OR user triggers explicit exit (OOC meta)
Guard:          none — user can always exit
Action:         Set session.active_experience = null
               Set session.entry_mode = "character"
               Reset session.active_experience fields to default
               Clear experience-specific knowledge entries
Postcondition:  session.active_experience == null
```

### LOCATION_CHANGE
```
Precondition:  session exists
Trigger:        User moves to new context (explicit or implicit)
Guard:         New location exists in entity store
               OR new location inferred from conditional load set
Action:        Set session.current_location = <new context_id>
               Update session.present_characters per entity.context.present_characters
               Trigger conditional load for new context
Postcondition:  session.current_location reflects new location
```

## Conflict Resolution Hook

E_18.1 only defines the agreement on conflict, not the enforcement details covered in E_17.4. Rules are applied during compilation.

| Conflict Type | Resolution |
|---------------|------------|
| Experience overrides identity | **REJECTED** — Identity is Level 1, immutable |
| Experience modifies behavior | **ALLOWED** — Behavior overlay, not identity change |
| Two experiences active | **REJECTED** — Guard prevents SCENARIO_START when experience != null |
| Visual contradicts character record | **Visual wins** — Visual Authority is Level 5, always applies |
| Knowledge entry contradicts entity | **Entity wins** — repository is SSoT |
| User claims override canon | **REJECTED** — ADR-000 runtime baseline |

## Schema Extension Protocol

When a new entity type is introduced:

1. **Add type to Entity Type Enum** (§Entity Type Enum)
2. **Define new repository object schema** in Layer 1
3. **Define consumed fields** for runtime object schema in Layer 2
4. **No Layer 3 action needed** — Adapter compiles from Layer 2 generically
5. **Update this document** with the new type

**Historical note:** This protocol is designed for future expansion (faction, species, vehicle) without breaking Layer 2/Layer 3 or refactoring the pipeline.

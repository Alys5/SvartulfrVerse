## Purpose

Define the **modular script and lorebook architecture** for the SvartúlfrVerse runtime. This system centralizes shared knowledge into reusable modules, eliminating per-bot duplication while enabling universe-scale (and future multiverse-scale) consistency.

**This is NOT a new engine.** This is the JanitorAI-specific implementation layer that lives INSIDE the pipeline defined by E_18.0 and E_18.0A. The compiler generates these artifacts; JanitorAI executes them.

---

## Architecture Overview

### The Problem

Without modular scripts, every bot export duplicates:
- World rules (human-only baseline, contemporary setting) → repeated in 12 System Prompts
- Family relationships (Douglas-Bloodmoon structure) → repeated in 12 Lorebooks
- Core mechanics (emotion engine, behavior patterns) → repeated in 12 Personality blocks
- Speech conventions (dynasty naming, register patterns) → repeated in 12 Example Dialogues

**Result:** Token waste, inconsistency risk, maintenance nightmare. Change the family structure → update 12 bots manually.

### The Solution: Layered Module Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    MODULAR SCRIPT LAYERS                     │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Layer 1: FOUNDATION                                         │
│  ├── CORE_ENGINE script (mechanics, emotion, behavior)       │
│  └── WORLD_*.js scripts (setting-specific rules)             │
│                                                              │
│  Layer 2: DYNASTY                                            │
│  ├── FAMILY_*.lorebook scripts (shared family knowledge)     │
│  └── CHARACTER_*.lorebook scripts (per-member detail)        │
│                                                              │
│  Layer 3: PER-BOT                                            │
│  ├── Bot-specific personality.md                              │
│  ├── Bot-specific scenario.md                                 │
│  ├── Bot-specific example_dialogue.md                         │
│  └── Bot-specific experience lorebooks                       │
│                                                              │
│  Cross-Cutting:                                               │
│  ├── Inclusion Groups (priority/traffic control)             │
│  └── Trigger Chains (inter-module communication)             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## Module Registry

### Current Module Manifest

| Module ID | Type | Target | Scope | Description |
|-----------|------|--------|-------|-------------|
| `MOD-CORE` | Advanced Script | All bots | Universal | Core engine: emotion state, behavior rules, response formatting, conversation mechanics |
| `MOD-WORLD-MODERN` | Advanced Script | All bots | Universal (contemporary) | Modern world rules: technology level, social norms, surveillance state, human-only baseline |
| `MOD-FAMILY-DB` | Lorebook Script | Dynasty members | Douglas-Bloodmoon family | Shared family knowledge: structure, relationships, key members, naming conventions, history |
| `MOD-CHAR-*` | Lorebook Script | Individual bot | Per-character | Per-member deep knowledge: personal arc, secrets, speech specifics, individual relationships |

### Naming Convention

```
MOD-[TYPE]-[SCOPE]

Types:
  CORE    → Core engine mechanics (Advanced Script)
  WORLD   → World/scenario context (Advanced Script)
  FAMILY  → Family/dynasty knowledge (Lorebook Script)
  CHAR    → Character-specific knowledge (Lorebook Script)
  EXP     → Experience/arc-specific knowledge (Lorebook Script)

Scope:
  Universal → Applies to all bots in the universe
  Family    → Applies to dynasty members only
  Character → Applies to one specific bot
```

## Layer 1: Foundation Modules

### 1.1 — MOD-CORE (Core Engine Script)

**Type:** Advanced Script
**Target:** All bots in the universe
**Description:** Shared behavioral mechanics that every SvartulfrVerse bot must follow

**Contents:**
- Emotion state tracking (current_mood, intensity, transition triggers)
- Response formatting rules (paragraph structure, POV consistency, markdown usage)
- Conversation mechanics (turn-taking, internal monologue formatting, action boundaries)
- OOC handling rules (command recognition, meta-discussion boundaries)
- Prohibited behaviors (never speak for {{user}}, never break character, never generate user actions)

**Why shared:** These are UNIVERSE-WIDE behavioral rules. Every bot in SvartulfrVerse must follow the same conversation mechanics regardless of character.

```javascript
// MOD-CORE structure (conceptual, not implementation)
{
  "module_id": "MOD-CORE",
  "type": "advanced_script",
  "scope": "universal",
  "triggers": ["always_active"],
  "rules": {
    "emotion_engine": {
      "track_state": true,
      "intensity_range": [0, 10],
      "transition_on": ["major_event", "user_conflict", "intimacy_moment"]
    },
    "response_format": {
      "perspective": "third_person_limited",
      "paragraph_structure": "action_dialogue_internal",
      "max_paragraphs": 4
    },
    "conversation_mechanics": {
      "never_speak_for_user": true,
      "never_assume_user_thoughts": true,
      "action_format": "*asterisks*",
      "internal_monologue_format": "_italics_",
      "dialogue_format": "\"quotes\""
    }
  }
}
```

### 1.2 — MOD-WORLD-* (World Context Scripts)

**Type:** Advanced Script (one per world setting)
**Target:** All bots compatible with the world setting
**Description:** World-specific rules, technology, social norms, and constraints

**Current Worlds:**
| Module ID | World | Status |
|-----------|-------|--------|
| `MOD-WORLD-MODERN` | Contemporary Los Angeles (2020s) | Active |
| `MOD-WORLD-HISTORICAL` | Future: pre-contemporary historical | Reserved |

**MOD-WORLD-MODERN Contents:**
- Technology level (smartphones, internet, AI assistants exist)
- Social norms (contemporary LA culture, entertainment industry presence)
- Surveillance infrastructure (Erik's PMC network, biometric monitoring)
- PMC-free zones (The Verve, off-grid locations)
- Cultural landscape (underground music scene, university culture, corporate elite)
- Human-only baseline enforcement (no supernatural elements per ADR-000)

```javascript
// MOD-WORLD-MODERN structure (conceptual)
{
  "module_id": "MOD-WORLD-MODERN",
  "type": "advanced_script",
  "scope": "world_contemporary",
  "world_id": "W_Contemporary",
  "triggers": ["always_active"],
  "rules": {
    "technology_level": "contemporary_2020s",
    "social_norms": "los_angeles_contemporary",
    "surveillance_state": "active_in_douglas_spaces",
    "pmc_free_zones": ["L_VerveLounge", "off_grid_locations"],
    "species_baseline": "human_only",
    "cultural_contexts": ["entertainment_industry", "underground_music", "corporate_elite"]
  }
}
```

**Extensibility:** Future worlds (historical, alternate timeline, multiverse variant) each get their own MOD-WORLD-* script. Multiple worlds can coexist in a multiverse scenario — each bot binds to its world-specific module.

## Layer 2: Dynasty Modules

### 2.1 — MOD-FAMILY-DB (Douglas-Bloodmoon Family Script)

**Type:** Lorebook Script
**Target:** All dynasty members (currently 12 characters: Wulfnic, Nixara, Erik, Malachia, Noah, Jasper, Alyssa, Logan, Edric + 3 outsiders)
**Description:** Shared family knowledge that every dynasty bot must know

**Contents:**
- Family tree structure (parent-child, marriages, derived relationships)
- Key family members (name, age, role, brief description per member)
- Dynasty naming conventions (Douglas-Bloodmoon surname rules from F_Surname_Authority)
- Family history highlights (union origin, historical events, business empire)
- Current family dynamics (Erik's patriarch role, Logan as safe haven, next-gen tensions)
- Internal family protocols (response to threats, loyalty expectations, public vs. private behavior)
- Shared locations (Douglas Estate as family seat)

**Why shared:** Every Douglas-Bloodmoon member shares this baseline. MOD-FAMILY-DB provides the common family grammar. Individual CHARACTER modules add personal depth.


## Layer 3: Per-Bot Composition

### 3.1 — Bot Assembly Matrix

Each bot's export package is assembled from module layers + bot-specific overrides:

```
Bot_Jasper/
├── personality.md              ← COMPILED: Layer 1 (CORE) rules + Layer 2 (FAMILY) identity + Bot-specific voice
├── scenario.md                 ← COMPILED: Layer 2 (WORLD) context + Bot-specific active scenario
├── initial_message.md          ← COMPILED: Bot-specific first beat
├── example_dialogue.md         ← COMPILED: Bot-specific speech demonstration
├── system_prompt.md            ← COMPILED: Layer 1 (CORE) rules + Layer 2 (WORLD) constraints + Bot-specific
│
├── scripts/
│   ├── MOD-CORE.js             ← SHARED: Core engine (linked, not duplicated)
│   ├── MOD-WORLD-MODERN.js    ← SHARED: World rules (linked, not duplicated)
│   └── (optional state tracker)
│
└── lorebooks/
    ├── MOD-FAMILY-DB.json      ← SHARED: Douglas-Bloodmoon knowledge (linked)
    ├── MOD-CHAR-JASP.json      ← SHARED: Jasper-specific knowledge (linked)
    └── MOD-EXP-DJF.json        ← TRIGGERED: DJ Frequency experience (bot-specific)
```

### 3.2 — Module Linking vs. Duplication

**Rule:** Modules are LINKED (referenced by ID in the bot's config), NOT duplicated into each bot's export package.

```json
// Bot configuration (conceptual)
{
  "bot_id": "C_Jasper",
  "modules": {
    "core": "MOD-CORE",
    "world": "MOD-WORLD-MODERN",
    "family": "MOD-FAMILY-DB",
    "character": "MOD-CHAR-JASP",
    "experiences": ["MOD-EXP-DJF"]
  },
  "overrides": {
    "personality_append": "... Jasper-specific additions ...",
    "system_prompt_append": "... Experienced-specific additions ..."
  }
}
```

**Implication:** When MOD-FAMILY-DB is updated (e.g., new family member added), ALL dynasty-linked bots automatically reflect the change. Zero per-bot maintenance.

## Inclusion Group Architecture

### Group Definitions

Inclusion groups prevent conflicting activations across modules:

| Group ID | Members | Selection Mode | Purpose |
|----------|---------|----------------|---------|
| `grp-behavior` | MOD-CORE.behavior_rules | One-at-a-time | Core behavior patterns |
| `grp-emotion` | MOD-CORE.emotion_engine | One-at-a-time | Emotional state tracking (EXCLUSION prevents multiple emotions simultaneously) |
| `grp-family-overview` | MOD-FAMILY-DB.entries | All-active | Family knowledge (all relevant entries inject simultaneously) |
| `grp-character-detail` | MOD-CHAR-*.entries | All-active | Character knowledge (all relevant entries inject) |
| `grp-world-context` | MOD-WORLD-*.entries | All-active | World knowledge (all relevant entries inject) |
| `grp-conflict-resolution` | MOD-CORE.authority_rules | Priority-based | Authority hierarchy resolution conflicts |

### Traffic Control Rules

```
Activation Flow:
1. Conversation message scanned for keywords
2. Matching entries collected from all modules
3. Inclusion Groups filter:
   a. Entries within same group → apply selection mode (one-at-a-time vs. all-active)
   b. Entries across groups → priority ordering (System Prompt rules highest)
4. Token budget check:
   a. If within budget → inject all surviving entries
   b. If over budget → drop Optional-priority first, then Supportive
   c. NEVER drop Critical-priority entries
5. Entries injected in priority order into prompt assembly
```

## Multiverse Extension Protocol

The modular architecture supports future multiverse scenarios:

```
Current Universe (SvartulfrVerse-LA):
├── MOD-CORE (universal mechanics)
├── MOD-WORLD-MODERN (contemporary LA)
├── MOD-FAMILY-DB (Douglas-Bloodmoon)
└── MOD-CHAR-* (12 characters)

Future Universe (e.g., Historical):
├── MOD-CORE (same mechanics, versioned)
├── MOD-WORLD-HISTORICAL (1666/1700s rules)
├── MOD-FAMILY-BLOODMOON-PREMERGE (pre-union Bloodmoon clan)
└── MOD-CHAR-* (ancestor characters)

Future Universe (e.g., Alternate):
├── MOD-CORE (forked mechanics)
├── MOD-WORLD-ALTERNATE (alternate rules)
├── MOD-FAMILY-* (new dynasty)
└── MOD-CHAR-* (new characters)
```

**Module versioning:** Each module carries a `universe_id` field. The compiler loads modules matching the target universe. Cross-universe references use explicit universe tagging.
